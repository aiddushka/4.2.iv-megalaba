# МегаЛаба: умная теплица (IoT Greenhouse)

## 1. Общее описание

Проект реализует “умную теплицу” как систему из:
- Backend-сервиса (Python, FastAPI, PostgreSQL) — хранит данные, обеспечивает API, управляет состояниями устройств и правилами автоматизации.
- Двух веб-сайтов:
  - Dashboard (сайт №1) — отображает состояние теплицы и устройств, позволяет администратору “установить” зарегистрированные устройства на доску и управлять актуаторами.
  - Device Configurator (сайт №2) — используется для регистрации и первичной настройки устройств (условно “ноутбук/флешка”), до привязки к теплице.
- Эмуляторов устройств (Python-скрипты) — генерируют показания датчиков и имитируют поведение актуаторов, взаимодействуя с backend через HTTP API.
- Docker-инфраструктуры — поднимает backend, PostgreSQL и фронтенды.

Логика “физического устройства” моделируется через статусы:
- `registered` / `unassigned` — устройство зарегистрировано конфигуратором, но ещё не “установлено” в теплице (не отображается как установленное).
- `assigned` — устройство установлено на доску (получило локацию и/или владельца) и начинает полноценно участвовать в отображении/автоматизации.

Цель проекта для мегалабы: показать полный цикл IoT-системы “регистрация -> настройка -> установка на доску -> телеметрия -> команды -> автоматизация”, с акцентом на протоколы общения и безопасность.

## 2. Участники и роли

### Пользователи (люди)
- `admin` — администратор/оператор. Видит и управляет всеми устройствами, в том числе “не установленными”, выполняет `assign`.
- `user` — обычный пользователь (возможен сценарий с несколькими пользователями и изоляцией по владельцам/тенантам).

### Устройства (агенты/агентные устройства)
Устройство в системе описывается сущностью `Device` с `device_uid` и типом (датчик/актуатор/контроллер).

Для лабораторной достаточно следующих типов:
- Датчики:
  - `TEMP_SENSOR` — температура воздуха.
  - `HUMIDITY_AIR_SENSOR` — влажность воздуха.
  - `HUMIDITY_SOIL_SENSOR` — влажность почвы.
  - `LIGHT_SENSOR` — освещённость.
- Актуаторы:
  - `IRRIGATION_ACTUATOR` — полив.
  - `HEATER_ACTUATOR` — подогрев/обогрев.
  - `VENTILATION_ACTUATOR` — проветривание/вентиляция.
  - `LIGHT_ACTUATOR` — освещение.
- Опционально:
  - `CONTROLLER` — “контроллер теплицы/шлюз”, который агрегирует данные и выдаёт команды (в данной архитектуре это может быть просто логическое устройство в БД).

## 3. Архитектура компонентов

### 3.1 Backend-сервис
Backend отвечает за:
- CRUD устройства (регистрация, установка/привязка, статусы).
- Приём телеметрии от датчиков (`/sensor-data/`).
- Управление актуаторами (`/actuators/control`), хранение их состояния (`/actuators/status`).
- Отдачу агрегированного состояния для Dashboard (`/dashboard/state`).
- Управление правилами автоматизации (`/automation/rules`) — логика “если условие по датчикам выполняется, то выполнить действие”.
- Безопасность: аутентификация и авторизация (пользователи и устройства) + изоляция пользователей.

### 3.2 Dashboard (сайт №1)
Dashboard визуализирует:
- Последние телеметрические данные датчиков.
- Текущее состояние актуаторов.
- Раздел для администратора: “неустановленные устройства” (то, что зарегистрировано через сайт №2).
- Действия администратора:
  - `assign` устройства (установка на доску, указание локации).
  - Команды актуаторам (кнопки ON/OFF в UI).

### 3.3 Device Configurator (сайт №2)
Конфигуратор обеспечивает:
- Регистрацию устройства: заполнение `device_uid`, типа, метаданных и желаемой локации (как “место на доске”).
- Отправку регистрационных данных в backend.
- Протокол взаимодействия “ноутбук/флешка -> backend -> устройство” в рамках лабораторного сценария:
  - В расширенной версии можно выдать устройству конфигурацию, включая сетевые параметры и секрет/ключ для подписи запросов.

### 3.4 Эмуляторы устройств
Эмуляторы реализуют поведение как:
- Сенсоры: периодически отправляют `value` на backend.
- Актуаторы: периодически отправляют команды или имитируют состояние, получая команды через API (в простейшей версии команда приходит через `/actuators/control`).

Эмуляторы нужны для демонстрации протокола и работы UI без реального железа.

## 4. Функциональные сценарии (end-to-end)

### 4.1 Добавление устройства (ваша “двухсайтовая” логика)
Шаги:
1. Физически установили устройство в нужном месте теплицы (в системе это ещё не отражено).
2. Подключились к сайту №2 — Device Configurator.
3. Отправили “регистрационные данные” устройства в backend:
   - `device_uid` (уникальный идентификатор),
   - `device_type` (тип датчика/актуатора),
   - `description` и `location_hint` (первичная привязка по смыслу).
4. Устройство попадает в список `unassigned`/`registered` в БД.
5. С сайта №1 (Dashboard) администратор открывает “неустановленные устройства”.
6. Администратор нажимает “Установить”, выбирает локацию/зону.
7. Backend переводит устройство в статус `assigned`.
8. После этого устройство отображается как установленное на доске и может участвовать в:
   - автоматизациях (правилам),
   - отображении состояния теплицы.

### 4.2 Телеметрия датчиков
1. Устройство-датчик по расписанию отправляет телеметрию на backend.
2. Backend:
   - проверяет подлинность устройства,
   - записывает показание в `sensor_data`,
   - (опционально в расширенной версии) запускает автоматизацию по поступившим данным.
3. Dashboard периодически запрашивает `/dashboard/state` или запрашивает новые данные для конкретных устройств.

### 4.3 Управление актуаторами
1. Backend принимает команду с Dashboard:
   - `device_uid` актуатора,
   - `action` (`ON` / `OFF`).
2. Backend сохраняет новое состояние актуатора в `actuators`.
3. Dashboard сразу отражает изменение состояния.
4. В расширенной версии устройство может подтверждать факт выполнения команды, отправляя статус (или “ack”) на backend.

### 4.4 Автоматизация и скрипты
Требование: “поддержка скриптов”.

На уровне проекта реализуется в виде:
- Правила (Automation Rules) хранятся в таблице `automation_rules`.
- Условия правила используют `sensor_type` и сравнение с порогом:
  - поддерживаемые операторы: `>`, `<`, `>=`, `<=` (для начала)
- Действие — какой `actuator_type` перевести в `ON` или `OFF`.

Исполнение:
- В минимальной версии автоматизация может выполняться:
  - при поступлении каждого `sensor_data` (event-driven),
  - либо периодическим фоновым циклом (polling).

Сценарий:
1. Пользователь/админ создаёт правило (UI или API).
2. Когда приходят новые данные датчика (например, влажность почвы падает ниже порога), backend вычисляет условия.
3. Если условие выполняется, backend записывает команду актуатору или напрямую меняет состояние.

Расширение “скриптов”:
- Вместо простых правил backend можно хранить исполняемую логику (например, шаблоны команд или ограниченный набор сценариев).
- Для безопасности и контрольности скрипты должны исполняться в песочнице и с whitelist API (в рамках мегалабы обычно достаточно правил).

## 5. Протоколы общения (что описать на “уровне протокола”)

Задача из пункта 3: “описать протоколы общения устройств” в понятной форме.

В рамках проекта можно описать соответствие по слоям OSI:
- Уровень 3 (Network): как устройства достигают backend по IP сети (Wi‑Fi/Ethernet/Zigbee шлюз).
- Уровень 4 (Transport): TCP (HTTPS/HTTP) или (опционально) MQTT поверх TCP.
- Уровень 5–6 (Session/Presentation): формат сообщений и сессии (в нашем HTTP JSON — это сериализация JSON и логика повторов/тайм-аутов).
- Уровень 7 (Application): прикладные протоколы:
  - REST API (HTTP + JSON),
  - формат телеметрии и команд,
  - схема аутентификации устройств.

Ниже — прикладной протокол (Application) для устройств.

### 5.1 Протокол телеметрии датчиков (Application Protocol)
Клиент (устройство):
- отправляет запрос `POST /sensor-data/`
- payload содержит значение сенсора
- обязательны поля подписи (см. раздел безопасности устройства)

Пример логической структуры (JSON):
- `device_uid`: строка идентификатора устройства
- `sensor_type` (опционально): тип сенсора, например `temperature`
- `value`: численное значение

### 5.2 Протокол управления актуаторами (Application Protocol)
Backend получает команду:
- `POST /actuators/control`

payload:
- `device_uid`
- `action`: `ON` / `OFF`
- `actuator_type` (опционально)

После выполнения устройство или эмулятор может:
- отправить статус на backend (в расширенной версии),
- либо backend хранит “последнее состояние” как результат команды.

### 5.3 Протокол статуса (Application Protocol)
Dashboard и эмуляторы используют:
- `GET /actuators/status` — вернуть состояния актуаторов.
- `GET /dashboard/state` — агрегированный снимок для UI.

## 6. Безопасность: аутентификация, авторизация, защита протоколов

Нужно закрыть пункты 2, 4, 5 из вопроса Никиты: basic authentication, устройства и изоляция пользователей.

### 6.1 Аутентификация пользователей сайтов (сайт №1/№2)
Рекомендуемый вариант для многопользовательской системы:
- Login/Registration пользователя в backend.
- Токены:
  - JWT (Bearer token) для Dashboard API.
- Пароли:
  - хранить в БД в виде хэшей (bcrypt/argon2).

Минимальная логика (для защиты по приложению):
- Dashboard:
  - доступ к операциям просмотра/управления закрывать по JWT.
  - доступ к “неустановленным” и `assign` только для `admin`.
- Device Configurator:
  - простой доступ можно сделать через HTTP Basic (как “engineering account”) поверх HTTPS.
  - или использовать отдельную роль/operator через JWT.

Если в мегалабе важен “Basic authentication”, то можно так:
- `/devices/register` доступен только по Basic Auth (username/password оператора).
- Это соответствует требованию “basic аутентификации на уровне приложения”.

### 6.2 Аутентификация устройств (подлинность запросов)
Критично: нельзя доверять `device_uid`, присланному с устройства без проверки.

Рекомендуемая схема для устройств:
- Каждому устройству выдаётся `device_secret` (или `private key`).
- Каждому запросу устройство добавляет подпись HMAC:
  - заголовки `X-Device-Uid`, `X-Timestamp`, `X-Nonce`, `X-Signature`.
  - `Signature = HMAC(device_secret, method + path + body + timestamp + nonce)`

Backend проверяет:
- существует ли устройство `device_uid`,
- совпадает ли HMAC,
- timestamp в допустимом окне (например ±5 минут),
- nonce не использовался ранее (anti-replay).

Результат:
- Даже если злоумышленник подменит device_uid, подпись не пройдёт.
- Даже если злоумышленник перехватит запрос, replay будет отклонён по nonce/timestamp.

### 6.3 Авторизация (кто что может)
После аутентификации:
- `admin` может видеть все устройства и выполнять `assign` для всех или всех “своих” теплиц.
- `user` может видеть и управлять только принадлежащими ему устройствами.

Реализация:
- в БД хранить `owner_id` или `tenant_id` (для каждого устройства).
- на каждый запрос фильтровать выборки по текущему пользователю.

### 6.4 Изоляция пользователей (multi-tenant)
Цель пункта 5: “продумать изоляцию пользователей”.

Базовый подход:
- Тенантность/владение устройствами:
  - устройства в таблице `devices` связаны с владельцем.
- Любые запросы на чтение/управление:
  - выполняются только в рамках владельца текущего пользователя.

Усиление (опционально, но красиво для защиты):
- PostgreSQL Row-Level Security (RLS).
- Audit log:
  - кто и когда регистрировал/назначал устройство,
  - кто отправлял команды актуаторам.

### 6.5 Ограничения и отказоустойчивость
Минимальные меры:
- Rate limiting на endpoints устройств:
  - `POST /sensor-data/` и `POST /actuators/control`.
- Idempotency:
  - можно добавить `X-Idempotency-Key` для команд (в расширении).
- Логи:
  - запись всех неуспешных попыток аутентификации устройств.

## 7. Модель данных (как хранить сущности)

Рекомендуемая схема (в терминах таблиц проекта):
- `users`
  - `id`
  - `username`
  - `hashed_password`
  - `is_admin`
- `devices`
  - `id`
  - `device_uid` (уникальный)
  - `device_type` (категория устройства)
  - `description`
  - `owner_id` (или tenant)
  - `location` (локация/зона на доске)
  - `status` (`registered/unassigned/assigned`)
  - (в расширении) `device_secret_hash` для подписи устройств
- `sensor_data`
  - `id`
  - `device_uid`
  - `sensor_type`
  - `value`
  - `created_at`
- `actuators`
  - `id`
  - `device_uid` (уникально в простейшем варианте)
  - `actuator_type`
  - `state` (`ON`/`OFF`)
- `automation_rules`
  - `id`
  - `name`
  - `sensor_type`
  - `condition`
  - `threshold`
  - `actuator_type`
  - `action`

Связи:
- `sensor_data.device_uid -> devices.device_uid`
- `actuators.device_uid -> devices.device_uid`
- `automation_rules` использует `sensor_type` и `actuator_type` как “типовые соответствия”.

## 8. API: что должно существовать (для описания Никите)

Группы endpoint’ов backend:

### 8.1 Auth
- `POST /auth/register`
- `POST /auth/login`

### 8.2 Devices (конфигуратор + admin assign)
- `POST /devices/register`
  - принимает `device_uid`, `device_type`, `description`, `location_hint`
  - добавляет запись в `devices` со статусом `unassigned`
- `GET /devices/unassigned`
  - отдаёт список устройств с `status=unassigned`
  - доступ только `admin` (и/или по tenant)
- `POST /devices/assign`
  - принимает `device_uid` и `location`
  - переводит в `assigned` и сохраняет локацию/владельца

### 8.3 Sensor data
- `POST /sensor-data/`
  - для датчиков: принимает `device_uid`, `value`, `sensor_type` (опционально)
- `GET /sensor-data/`
  - для диагностики/аналитики

### 8.4 Actuators
- `POST /actuators/control`
  - принимает команду `device_uid`, `action`, (опционально) `actuator_type`
- `GET /actuators/status`
  - список текущих `state`

### 8.5 Dashboard + Automation
- `GET /dashboard/state`
  - агрегирует последние sensor_data и состояния actuators
- `POST /automation/rules`
  - создание правила
- `GET /automation/rules`
  - список правил

## 9. Запуск и демонстрация (как защитить на практике)

Минимальный план демонстрации:
1. Поднять инфраструктуру через Docker.
2. Запустить эмуляторы датчиков и актуаторов.
3. Открыть Dashboard:
   - убедиться, что приходят данные в `/dashboard/state`.
4. Открыть Device Configurator:
   - зарегистрировать устройство нового `device_uid`.
5. На Dashboard открыть “неустановленные устройства”:
   - выполнить `assign`.
6. Показать автоматизацию:
   - создать правило и затем продемонстрировать реакцию (например, при влажности почвы ниже порога включается полив).

## 10. Что из этого уже является реализованным каркасом, а что требуется расширить

В репозитории уже есть базовые компоненты:
- Backend: FastAPI + эндпоинты устройств/данных и Docker.
- Эмуляторы: датчики (температура/влажность/свет) и часть актуаторов.
- Frontend: Dashboard (доска) и Device Configurator (регистрация устройства).

Для “полной защиты по безопасности” и “правильной мегалабы” рекомендуется довести:
- устройство-аутентификацию (HMAC/подписи запросов),
- доступы по ролям (JWT/Bearer для Dashboard, BasicAuth для Device Configurator),
- изоляцию пользователей по `owner_id/tenant_id`,
- анти-replay (nonce) и rate limiting на endpoints устройств,
- event-driven автоматизацию (с привязкой к sensor_type).

Эти пункты легко объяснить Никите как “проектная спецификация” и показать как цель дальнейшей доработки.
