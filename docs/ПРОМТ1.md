ЗАДАЧА:
Расширить систему управления устройствами, добавив новые поля для более детальной конфигурации. Сделать это в два этапа: сначала обновить бэкенд и базу данных, затем обновить оба фронтенда.

ЧАСТЬ 1. БЭКЕНД И БАЗА ДАННЫХ

1.1. В файле backend/app/models/device.py добавить следующие поля в модель Device:
- image_url (String, nullable=True) - ссылка на изображение устройства
- controller (String, nullable=True) - тип контроллера (Arduino Uno, ESP32 и т.д.)
- pin (Integer, nullable=True) - номер GPIO пина
- bus (String, nullable=True) - тип шины подключения (Digital, Analog, I2C, SPI, OneWire)
- bus_address (String, nullable=True) - адрес на шине (для I2C)
- components (JSON, nullable=True) - список дополнительных компонентов
- status (String, default="active") - состояние устройства (active/maintenance/offline)
- last_maintenance (DateTime, nullable=True) - дата последнего обслуживания
- maintenance_notes (String, nullable=True) - заметки по обслуживанию
- change_history (JSON, nullable=True) - история изменений конфигурации

1.2. В файле backend/app/schemas/device_schema.py создать следующие схемы:
- DeviceCreate (расширить существующую) - добавить все новые поля для регистрации
- DeviceUpdate (расширить существующую) - добавить поля для редактирования на главном сайте
- DeviceOut (расширить существующую) - добавить все поля для вывода
- DeviceAdminUpdate - отдельная схема для админских полей (status, maintenance_notes)

1.3. В файле backend/app/services/device_service.py обновить функции:
- register_device() - сохранять все новые поля из формы регистрации
- update_device() - добавить возможность обновления полей: description, location, status, last_maintenance, maintenance_notes
- добавить функцию log_change_history() - для автоматической записи изменений в change_history

1.4. В файле backend/app/api/devices.py обновить эндпоинты:
- POST /devices/register - принимать все новые поля
- PATCH /devices/{device_uid} - расширить для обновления статуса и полей обслуживания

1.5. Создать миграцию для добавления новых колонок в таблицу devices. Использовать SQLAlchemy или прямой SQL запрос.

ЧАСТЬ 4. ДОПОЛНИТЕЛЬНЫЕ ТРЕБОВАНИЯ

4.1. Все новые поля должны быть опциональными в БД (nullable) для обратной совместимости

4.2. Для изображений использовать простое текстовое поле image_url (ссылка на картинку), без реализации загрузки файлов на сервер

4.3. Поле components хранить как JSON массив строк в БД, на фронтенде работать как с массивом

4.7. Внешний вид: сохранить существующую стилизацию (тема с фоном #020617, #0f172a, акцентные цвета #22c55e и #22d3ee)
