# Сайты проекта (Dashboard и Device Configurator)

В проекте реализованы два веб-интерфейса, оба доступны через Nginx по HTTPS.

## 1. Dashboard (`frontend-dashboard`)

- URL: `https://localhost:8443/`
- Назначение: основной рабочий интерфейс для мониторинга и управления системой.

### Роли пользователей

- **Администратор**: управляет устройствами, назначает локации, редактирует настройки, создает связи датчик -> актуатор, выдает права работникам.
- **Работник**: имеет доступ к просмотру Dashboard при включенном флаге `can_view_dashboard`.

### Ключевые разделы

- главная доска с датчиками, актуаторами и состоянием устройств;
- список неустановленных устройств для активации;
- управление работниками и правами;
- карточка устройства: информация, управление, история изменений.

## 2. Device Configurator (`frontend-device-config`)

- URL: `https://localhost:8443/config/`
- Назначение: первичная регистрация IoT-устройства в системе до назначения на локацию.

### Что вводится при регистрации

- тип устройства (датчик или актуатор);
- контроллер (например, ESP32, Arduino, Raspberry Pi);
- пин/интерфейс подключения (Digital, Analog, I2C, SPI, OneWire);
- компоненты и описание;
- при необходимости - связь с другим устройством (обычно датчик -> актуатор).

## 3. Работа через Nginx и API

- внешний порт: `8443` (HTTPS);
- HTTP `8080` перенаправляется на HTTPS;
- API вызывается с фронтендов через префикс `/api`;
- внутренние Vite-порты (`5173`, `5174`) наружу не публикуются.

Примеры пользовательских запросов:

- `GET /api/dashboard/state`
- `POST /api/devices/register`
- `POST /api/devices/assign`
- `POST /api/actuators/control`

## 4. IoT-логика интерфейсов

- Dashboard отображает не только текущие телеметрические значения, но и устройства в состоянии ожидания первых данных.
- Device Configurator заполняет модель физического подключения устройства, что важно для обслуживания IoT-схемы в теплице.
- Связи между датчиками и актуаторами позволяют реализовать автоматизацию по пороговым значениям.
